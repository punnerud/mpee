brooom — kost-matching mot Vroom (runde 3)
================================================

Ny regel:
  - Kost MÅ ned mot Vroom-paritet — viktigst.
  - Tid kan øke. Mål: holde oss under Vroom's tid hvis mulig (vi har 30-40×
    headroom på store instanser), men ikke kritisk.
  - RSS bør holdes lavt; ikke et hardt krav.

[x] 3. Re-aktiver or-opt med reversal

[x] 4. Default multi-start K=8
   Strategi: bruk all CPU for diversitet — seed 0 deterministisk + 1..7
             shuffler ventende tasks innen prioritetsklasse.
   Resultat:  Stort kostvinn på N=50/100/250 (~5-10% over single-start).
              Tidskostnad: ~3-4× single-start men fortsatt under Vroom på
              N≤500. Ingen regresjon på noen størrelse.
   Verdikt:   AKTIVERT (default=8).

[x] 5. Iterated Local Search (ILS) med destroy-and-repair kicks
   Strategi: etter LS-konvergens, fjern frac% av tasks tilfeldig,
             reinsert via cheapest-feasible (try_insert_single +
             evaluate_route validering), re-LS. Track best-ever.
   Resultat:  Default ils_iters=30, kick=0.4. På N=100/250 ga dette
              vinneren (-1.0% / -0.8% vs Vroom). På N=500/1000 marginal
              forbedring uten å slå Vroom.
   Verdikt:   AKTIVERT (default ils_iters=30, ils_kick_size=0.4).

[x] 5b. LRU-cache for evaluate_route (thread-local + global epoch)
   Strategi: HashMap<(vehicle_hash, steps_hash), Result<RouteMetrics>>
             per worker-thread; global AtomicU64 epoch som solver bumper
             ved start. Workers sjekker epoch og tømmer ved stale.
   Resultat:  Modest hastighetsforbedring (~5-15% på ILS-tunge runs).
              Ingen kostendring (cache er korrekthetsbevarende).
   Verdikt:   AKTIVERT.

[-] 9. REINFORCE-vekter på ILS-task-utvalg (PyTorch-stil gradient)
   Strategi: hver task har en sample-vekt. ILS-kick velger n_kick tasks
             via Efraimidis-Spirakis (key = -ln(U)/w). Etter LS:
             multiplikativ bump/decay basert på sign(Δkost).
             Testet med α=0.15 (1.15/0.95) og α=0.05 (1.05/0.98).
   Resultat:  Begge varianter gir regresjon over hele linja:
              α=0.15: N=100 +1.3%, N=250 +1.0%, N=500 +1.2%, N=1000 +1.4%
              α=0.05: N=100 +0.9%, N=250 +1.9%, N=500 +1.8%, N=1000 +0.8%
              Årsak: ILS henter all sin verdi fra diversitet i kicks. Med
              bare 30 iterasjoner per variant er vi i utforsknings-regime,
              ikke utnyttelses-regime. REINFORCE konsentrerer kicks på
              "vanskelige" tasks, men disse er ofte vanskelige nettopp
              fordi de er trange — å fjerne dem oftere produserer bare
              flere unassigned, ikke bedre globale optima.
              Dette er klassisk explore-vs-exploit: 30 kicks er for få til
              at policy-konvergens lønner seg.
   Verdikt:   REVERTERT.
