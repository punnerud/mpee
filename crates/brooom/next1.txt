#
# ============================================================
# PENALTY-LS DESIGN
# ============================================================
#
# Kjerne-idé: tillat TW- og kapasitets-brudd under LS med en
# kost-straff som motivasjon. Lar LS hoppe gjennom "infeasible
# mellom-løsninger" til bedre feasible optima.
#
# ---- Megakernel-endringer (~300-400 LOC) ----
#
# Nye workgroup-arrays:
#   var<workgroup> wg_route_dist: array<i32, 512>;     // distanse per rute
#   var<workgroup> wg_route_tw_viol: array<i32, 512>;  // TW-brudd i seconds
#   var<workgroup> wg_route_cap_viol: array<i32, 512>; // capacity overrun
#
# Phase 1 (init): walk hver rute, beregn dist + TW-violation + cap-violation
#
# Phase 2a-g (operatorer): for hver kandidat-move:
#   - Compute dist delta (cheap, som før)
#   - Walk proposed routes, compute new TW-violations + cap-violations
#   - total_delta = dist_delta
#                 + W_tw * (new_tw_viol_total − cur_tw_viol_total)
#                 + W_cap * (new_cap_viol_total − cur_cap_viol_total)
#   - atomicMin på total_delta (ikke bare dist_delta)
#
# Phase 4 (apply): oppdater wg_route_* for berørte ruter
#
# Penalty-vekter (params/uniform):
#   - W_tw_init = 100, W_cap_init = 1000 (mild — utforskning)
#   - Ramp opp lineært over iterasjoner til 1e6 (hard — feasibility)
#   - I "exploration" fase: mange infeasible moves akseptert
#   - I "convergence" fase: kun feasible
#
# Sluttkontroll: hvis sluttløsning fortsatt infeasible (W_tw/W_cap ikke
# var høye nok): repair-fase som påfører høyere straff til feasible.
#
# ---- TW-violation-beregning ----
#
# I dag: walk route, if current_t > tw_end[here]: break (infeasible).
# Penalty-versjon: walk route, akkumuler max(0, current_t - tw_end[here])
# for hvert stopp. Det er totalt TW-brudd.
#
# Implementasjons-detalj: "current_t = max(current_t, tw_start[here])"
# håndteres som før (wait-time). "if current_t > tw_end[here]" blir til
# "violation += current_t - tw_end[here]; current_t = current_t".
# Vi clamper IKKE current_t — infeasibility propagerer fremover.
#
# ---- Estimat per komponent ----

# ---- Tidligere (A)/(B) beskrivelser (historisk) ----
#
# ---- (A) swap-star (Vidal 2014, "ESPPRC swap-star") ----
#
# Generaliserer exchange: bytter (a i rute A, b i rute B), MEN
# *omposisjonerer* hver oppgave til den beste plassen i sin nye rute.
#
# Per (a, b) granular-par (rA != rB):
#   - removal_a = dist(prev_iA, a) + dist(a, next_iA) − dist(prev_iA, next_iA)
#   - removal_b = dist(prev_pB, b) + dist(b, next_pB) − dist(prev_pB, next_pB)
#   - For hver kandidat-posisjon p_a i B' (B etter at b er fjernet):
#       insertion_a_at_p = dist(prev, a) + dist(a, next) − dist(prev, next)
#       hold best_a (laveste insertion-kost)
#   - Tilsvarende best_b i A'
#   - delta = best_a + best_b − removal_a − removal_b
#
# Kandidat-posisjoner: full slot (~30 stops per rute) er for dyrt.
# Granular-trick: kun posisjoner *adjacent* til a's K-naboer i B.
# Eks: hvis a's tre nærmeste naboer i B er på posisjon 4, 8, 12,
# da prøver vi p_a ∈ {3,4,5, 7,8,9, 11,12,13} — 9 kandidater i stedet
# for hele rute-slot. O(K) per par.
#
# Total kompleksitet: O(N × K × K) per LS-iter. For N=1000, K=50:
# 2.5M evalueringer / 1024 tråder = ~2500 per tråd × 100 ops = 250K
# ops/tråd/iter. Estimert ~30-50 ms/iter ved N=1000 (mot 3 ms uten
# swap-star). Trade-off: ~10× tregere per iter, men <2% færre iter.
#
# Apply-fase er nontriviell: må committe 6 verdier (rA, iA, rB, iB,
# p_a_new, p_b_new). Pakk to par i én atomic u32:
#   wg_best_route = (rA << 16) | rB
#   wg_best_i     = (iA << 16) | iB
#   wg_best_j     = (p_a_new << 16) | p_b_new
#
# Apply-rekkefølge:
#   1. Lagre a og b
#   2. Fjern a fra A (shift venstre)
#   3. Fjern b fra B (shift venstre)
#   4. Sett inn b i A på p_b_new (shift høyre fra p_b_new)
#   5. Sett inn a i B på p_a_new (shift høyre fra p_a_new)
#
# Estimert innsats: ~250 linjer WGSL (søk: 150, apply: 100). 1-2 dager
# inkl. validering mot CPU-evaluate_route på 50+ ruter.
#

## Bakgrunn

I dag opererer brooom GPU-megakernelen i to moduser:

1. **Matrix-mode**: hele N×N i32 avstandsmatrise lastet til GPU. Praktisk
   tak ~N=16K (1 GiB buffer-grense). For N=50K ville matrisen vært 10 GB.
2. **Coord-mode** (nylig): coords lastes opp som f32×2 per node, avstander
   regnes ut on-demand i shader via `sqrt(dx²+dy²) × 100`. Fungerer på
   N=50K, men kun for Euclidean. Klarer ikke å bruke OSRM-veiavstander.

Begrensningen: for ekte produksjon (OSRM-baserte kjøreavstander, ikke
fugleflukt) trenger vi enten:
- Full N×N matrise (umulig for N≥16K pga minne/båndbredde)
- En *sparse* matrise med kun de avstandene LS-løkken faktisk bruker

## Idéen: streaming + sparse K-nabo-cache

### Kjerne-innsikt

Granular K=50 LS bruker bare **K=50 nærmeste naboer per node** for
relocate/exchange. Det er N×K = 2.5M avstander for N=50K — ikke N² = 2.5G.
2000× mindre minne, 2000× færre OSRM-kall.

### Datastruktur

```
sparse_cache[node_id]  →  [(neighbour_id, distance); K]
```

For N=50K, K=50: 50000 × 50 × 8 bytes = **20 MB** (mot 10 GB for tett).

### Distanse-lookup i shader

```wgsl
fn dist_at(a: u32, b: u32) -> i32 {
    // 1. Linear-scan i a's K-naboer (kun 50 ops)
    for (var k: u32 = 0u; k < K; k = k + 1u) {
        let entry = sparse_cache[a * K + k];
        if (entry.neighbour == b) {
            return entry.distance;
        }
    }
    // 2. Fallback: Euclidean fra coords
    let ax = coords[a * 2u];   let ay = coords[a * 2u + 1u];
    let bx = coords[b * 2u];   let by = coords[b * 2u + 1u];
    return i32(sqrt((ax-bx)*(ax-bx) + (ay-by)*(ay-by)) * 100.0);
}
```

Granular-baserte operatorer (relocate, exchange) treffer alltid sparse
cache — kjent veiavstand. 2-opt innen samme rute treffer sparse cache når
to noder i ruten er K-naboer, fallback til Euclidean ellers. Siden LS
fortrinnsvis går etter naboskap, vil "stupide" moves som krever fallback
nesten aldri være lovende uansett.

### Streaming-pipeline

Pseudo-kode for byggsteget:

```
sent = {depot}
emit(depot)                            # første frø
while |sent| < N:
    candidate = argmax_{c ∉ sent} min_{s ∈ sent} dist(c, s)  # fjernest fra eksisterende sett
    emit(candidate)
    sent.add(candidate)

    if |sent| >= warmup_threshold (e.g. 2000):
        request_knn(candidate, K=50)   # OSRM table call: candidate vs alle eksisterende
        sparse_cache[candidate] = top-K nærmeste
        # Også oppdater eldres K-naboer hvis candidate er nærmere
        for old in sparse_cache:
            if dist(candidate, old) < old.K_th:
                insert_sorted(sparse_cache[old], candidate)
                evict_if_overflow(K)

        if first_time: start_ls_thread()    # LS kan starte parallelt
```

**Farthest-point seeding** (maximin-rekkefølge):
- Garanterer at de første K%×N punktene dekker hele rommet
- Hvert nytt punkt har minst mulig redundans
- Når ~2000 punkter er på plass, har granular-tabellen god "skjelett"-
  dekning av domene-rommet — LS kan starte før resten er ferdig

## Fordeler

### Minne / båndbredde

| Aspekt | Tett N² | Sparse K-nabo |
|---|---|---|
| GPU-buffer (N=50K) | 10 GB | 20 MB |
| OSRM table-kall | 50000² = 2.5G | 50000 × 50 = 2.5M |
| Bandwidth | 10 GB | 20 MB |
| Tid å bygge matrise | minutter-timer | strømmet, ingen kald start |

500× reduksjon i minne, 1000× færre nettverkskall, **null kald-start-
forsinkelse** — LS kan starte etter ~K%×N punkter.

### Korrekthet

- Granular-operatorer (relocate, exchange) bruker *alltid* ekte
  veiavstander fra OSRM
- 2-opt bruker Euclidean for fjerne par. Filtrerer ut implausible moves
  som ikke ville vært valgt uansett (de er for langt fra hverandre til at
  forbedringen blir betydelig)

### Dynamisk forfining

Hvis Euclidean-fallback identifiserer en lovende 2-opt-move (Δ < terskel),
kunne vi spørre OSRM live for å bekrefte før commit. På den måten:
- LS kjører raskt på estimater
- OSRM kalles kun for de få avgjørelsene som faktisk betyr noe
- Total OSRM-bruk kan reduseres til **<1% av en tett matrise**

### Skalering

Sparse cache + coord-mode fallback enables:
- N=100K: 50 MB sparse cache, 1.6 MB coords. Bare ~30s OSRM-tid for KNN-
  bygg (parallelle table-kall)
- N=500K: 250 MB cache, 8 MB coords. Stadig innenfor M3 GPU-minne

Praktisk øvre grense flyttet fra **N≈16K → N≈500K**.

### CPU-side (Rust)

```rust
struct SparseCache {
    /// Per node: K (neighbour_id, distance) pars
    entries: Vec<(u32, i32)>,   // n × K, flat
    k: usize,
}

impl SparseCache {
    fn new(n: usize, k: usize) -> Self;
    fn insert(&mut self, src: u32, dst: u32, dist: i32);
    fn into_gpu_buffers(self) -> (Vec<u32>, Vec<i32>);  // splitt for GPU
```

To nye buffers i megakernel:

### Pipeline-arkitektur

```
[OSRM] ───stream─→ [SparseCache builder] ───→ [GPU: upload coords + sparse]
                            │
                            └─ trigger LS-start ved warmup_threshold
                                            │
                                            ▼
                              [megakernel LS] ──→ Solution
```

## Tradeoffs / risiko

1. **Cache-miss-rate i 2-opt**: ukjent hvor ofte LS prøver edges som ikke
   er K-naboer. Trolig <5% siden granular eliminerer fjerne kandidater,
   men må måles.
2. **OSRM throughput**: hvor raskt klarer OSRM N×K table-kall? Typisk
   1000 punkter/sek per kjerne → 50s for N=50K. Akseptabelt.
3. **Eldre nodes må oppdatere K-listen**: når et nytt punkt kommer inn
   som er nærmere enn deres K-te nabo. Krever sorterings-insertion.
   Amortisert O(N×K log K) total — overkommelig.
4. **Asymmetri**: OSRM-avstander er ikke symmetriske (envegskjøring).
   Sparse cache bør lagre begge retninger separat — fordobler minne men
   fortsatt 1000× mindre enn tett.

## Estimert innsats

| Fase | LOC | Tid | Effekt |
|---|---|---|---|
| Sparse-cache i shader | ~150 | 1 dag | N=50K → 20 MB, riktig OSRM-avstander |
| Farthest-point seeder | ~80 | ½ dag | Smartere KNN-rekkefølge |
| OSRM streaming-klient | ~200 | 1 dag | Live KNN-kall |
| Progressiv LS-start | ~100 | ½ dag | Skjul OSRM-latency bak LS |
| Dynamisk OSRM-verify for 2-opt | ~80 | ½ dag | Reduserer OSRM-kall ~99% |
| **Total** | **~610** | **3-4 dager** | N=50K på minutter i stedet for timer |

## Anbefalt rekkefølge

1. **Sparse-cache i shader** først — gir umiddelbar gevinst for N≥20K,
   selv med Euclidean-bygd sparse (uten OSRM)
2. **Farthest-point seeder** — billig, gjør benchmarks mer realistiske
3. **OSRM streaming-klient** — den faktiske I/O-integrasjonen
4. **Progressiv LS-start** — perf-vinn for produksjon
5. **Dynamisk verify** — siste finpussning, kan vente

Den første alene flytter taket fra N=16K til N=500K. Resten er
optimaliseringer for å holde OSRM-belastningen lav i produksjon.
