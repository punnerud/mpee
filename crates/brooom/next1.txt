# ============================================================
# PENALTY-LS DESIGN
# ============================================================
#
# Core idea: allow TW and capacity violations during LS with a
# cost penalty as motivation. Lets LS hop through "infeasible
# intermediate solutions" to better feasible optima.
#
# ---- Megakernel changes (~300-400 LOC) ----
#
# New workgroup arrays:
#   var<workgroup> wg_route_dist: array<i32, 512>;     // distance per route
#   var<workgroup> wg_route_tw_viol: array<i32, 512>;  // TW violation in seconds
#   var<workgroup> wg_route_cap_viol: array<i32, 512>; // capacity overrun
#
# Phase 1 (init): walk each route, compute dist + TW-violation + cap-violation
#
# Phase 2a-g (operators): for each candidate move:
#   - Compute dist delta (cheap, as before)
#   - Walk proposed routes, compute new TW-violations + cap-violations
#   - total_delta = dist_delta
#                 + W_tw * (new_tw_viol_total − cur_tw_viol_total)
#                 + W_cap * (new_cap_viol_total − cur_cap_viol_total)
#   - atomicMin on total_delta (not just dist_delta)
#
# Phase 4 (apply): update wg_route_* for affected routes
#
# Penalty weights (params/uniform):
#   - W_tw_init = 100, W_cap_init = 1000 (mild — exploration)
#   - Ramp up linearly over iterations to 1e6 (hard — feasibility)
#   - In "exploration" phase: many infeasible moves accepted
#   - In "convergence" phase: only feasible
#
# Final check: if the final solution is still infeasible (W_tw/W_cap were
# not high enough): repair phase that applies higher penalty until feasible.
#
# ---- TW-violation computation ----
#
# Today: walk route, if current_t > tw_end[here]: break (infeasible).
# Penalty version: walk route, accumulate max(0, current_t - tw_end[here])
# for each stop. That is total TW violation.
#
# Implementation detail: "current_t = max(current_t, tw_start[here])"
# is handled as before (wait-time). "if current_t > tw_end[here]" becomes
# "violation += current_t - tw_end[here]; current_t = current_t".
# We do NOT clamp current_t — infeasibility propagates forward.

#    d. If offspring is feasible AND better than the population's worst-diverse

# 4. Return best in population at time budget.

# ---- What we already have on GPU ----

#   ✓ GpuPopulation (pop_size, fleet, matrix on GPU)

#    - Drop tasks that cannot be reinserted feasibly (mark the trajectory)

# ---- (A) swap-star (Vidal 2014, "ESPPRC swap-star") ----
#
# Generalizes exchange: swaps (a in route A, b in route B), BUT
# *re-positions* each task to the best slot in its new route.
#
# Per (a, b) granular pair (rA != rB):
#   - removal_a = dist(prev_iA, a) + dist(a, next_iA) − dist(prev_iA, next_iA)
#   - removal_b = dist(prev_pB, b) + dist(b, next_pB) − dist(prev_pB, next_pB)
#   - For each candidate position p_a in B' (B after b is removed):
#       insertion_a_at_p = dist(prev, a) + dist(a, next) − dist(prev, next)
#       keep best_a (lowest insertion cost)
#   - Similarly best_b in A'
#   - delta = best_a + best_b − removal_a − removal_b
#
# Candidate positions: the full slot (~30 stops per route) is too expensive.
# Granular trick: only positions *adjacent* to a's K-neighbors in B.
# E.g.: if a's three nearest neighbors in B are at positions 4, 8, 12,
# then we try p_a ∈ {3,4,5, 7,8,9, 11,12,13} — 9 candidates instead
# of the full route slot. O(K) per pair.
#
# Total complexity: O(N × K × K) per LS iter. For N=1000, K=50:
# 2.5M evaluations / 1024 threads = ~2500 per thread × 100 ops = 250K
# ops/thread/iter. Estimated ~30-50 ms/iter at N=1000 (vs 3 ms without
# swap-star). Trade-off: ~10× slower per iter, but <2% fewer iters.
#
# Apply phase is nontrivial: must commit 6 values (rA, iA, rB, iB,
# p_a_new, p_b_new). Pack two pairs into one atomic u32:
#   wg_best_route = (rA << 16) | rB
#   wg_best_i     = (iA << 16) | iB
#   wg_best_j     = (p_a_new << 16) | p_b_new
#
# Apply order:
#   1. Save a and b
#   2. Remove a from A (shift left)
#   3. Remove b from B (shift left)
#   4. Insert b into A at p_b_new (shift right from p_b_new)
#   5. Insert a into B at p_a_new (shift right from p_a_new)
#
# Estimated effort: ~250 lines WGSL (search: 150, apply: 100). 1-2 days
# incl. validation against CPU evaluate_route on 50+ routes.
#
# ---- (B) `--gpu` integration in solve_full ----
#

## Background

Today brooom's GPU megakernel operates in two modes:

1. **Matrix mode**: full N×N i32 distance matrix loaded to GPU. Practical
   ceiling ~N=16K (1 GiB buffer limit). For N=50K the matrix would be 10 GB.
2. **Coord mode** (recent): coords uploaded as f32×2 per node, distances
   computed on-demand in the shader via `sqrt(dx²+dy²) × 100`. Works on
   N=50K, but Euclidean only. Cannot use OSRM road distances.

The limitation: for real production (OSRM-based drive distances, not
straight-line) we need either:
- Full N×N matrix (impossible for N≥16K due to memory/bandwidth)
- A *sparse* matrix with only the distances the LS loop actually uses

## The idea: streaming + sparse K-neighbor cache

### Core insight

Granular K=50 LS only uses **K=50 nearest neighbors per node** for
relocate/exchange. That is N×K = 2.5M distances for N=50K — not N² = 2.5G.
2000× less memory, 2000× fewer OSRM calls.

### Data structure

```
sparse_cache[node_id]  →  [(neighbour_id, distance); K]
```

For N=50K, K=50: 50000 × 50 × 8 bytes = **20 MB** (vs 10 GB for dense).

### Distance lookup in shader

```wgsl
fn dist_at(a: u32, b: u32) -> i32 {
    // 1. Linear scan in a's K-neighbors (only 50 ops)
    for (var k: u32 = 0u; k < K; k = k + 1u) {
        let entry = sparse_cache[a * K + k];
        if (entry.neighbour == b) {
            return entry.distance;
        }
    }
    // 2. Fallback: Euclidean from coords
    let ax = coords[a * 2u];   let ay = coords[a * 2u + 1u];
    let bx = coords[b * 2u];   let by = coords[b * 2u + 1u];
    return i32(sqrt((ax-bx)*(ax-bx) + (ay-by)*(ay-by)) * 100.0);
}
```

Granular-based operators (relocate, exchange) always hit the sparse
cache — known road distance. 2-opt within the same route hits the sparse cache when
two nodes in the route are K-neighbors, otherwise falls back to Euclidean. Since LS
preferentially follows neighborhoods, "stupid" moves that require fallback
are almost never promising anyway.

### Streaming pipeline

Pseudo-code for the build step:

```
sent = {depot}
emit(depot)                            # first seed
while |sent| < N:
    candidate = argmax_{c ∉ sent} min_{s ∈ sent} dist(c, s)  # farthest from existing set
    emit(candidate)
    sent.add(candidate)

    if |sent| >= warmup_threshold (e.g. 2000):
        request_knn(candidate, K=50)   # OSRM table call: candidate vs all existing
        sparse_cache[candidate] = top-K nearest
        # Also update older nodes' K-neighbors if candidate is closer
        for old in sparse_cache:
            if dist(candidate, old) < old.K_th:
                insert_sorted(sparse_cache[old], candidate)
                evict_if_overflow(K)

        if first_time: start_ls_thread()    # LS can start in parallel
```

**Farthest-point seeding** (maximin order):
- Guarantees that the first K%×N points cover the whole space
- Each new point has minimum possible redundancy
- When ~2000 points are in place, the granular table has a good "skeleton"
  coverage of the domain space — LS can start before the rest is done

## Advantages

### Memory / bandwidth

| Aspect | Dense N² | Sparse K-neighbor |
|---|---|---|
| GPU buffer (N=50K) | 10 GB | 20 MB |
| OSRM table calls | 50000² = 2.5G | 50000 × 50 = 2.5M |
| Bandwidth | 10 GB | 20 MB |
| Time to build matrix | minutes-hours | streamed, no cold start |

500× memory reduction, 1000× fewer network calls, **zero cold-start
latency** — LS can start after ~K%×N points.

### Correctness

- Granular operators (relocate, exchange) *always* use real
  road distances from OSRM
- 2-opt uses Euclidean for distant pairs. Filters out implausible moves
  that would not have been chosen anyway (they are too far apart for the
  improvement to be significant)

### Dynamic refinement

If the Euclidean fallback identifies a promising 2-opt move (Δ < threshold),
we could query OSRM live to confirm before commit. This way:
- LS runs quickly on estimates
- OSRM is called only for the few decisions that actually matter
- Total OSRM usage can be reduced to **<1% of a dense matrix**

### Scaling

Sparse cache + coord-mode fallback enables:
- N=100K: 50 MB sparse cache, 1.6 MB coords. Only ~30s OSRM time for KNN
  build (parallel table calls)
- N=500K: 250 MB cache, 8 MB coords. Still within M3 GPU memory

Practical upper limit moved from **N≈16K → N≈500K**.

## Implementation sketch

### CPU side (Rust)

```rust
struct SparseCache {
    /// Per node: K (neighbour_id, distance) pairs
    entries: Vec<(u32, i32)>,   // n × K, flat
    k: usize,
}

impl SparseCache {
    fn new(n: usize, k: usize) -> Self;
    fn insert(&mut self, src: u32, dst: u32, dist: i32);
    fn into_gpu_buffers(self) -> (Vec<u32>, Vec<i32>);  // split for GPU
}

trait DistanceProvider {
    fn knn(&mut self, src_coord: (f64, f64), candidates: &[(u32, (f64, f64))], k: usize)
        -> Vec<(u32, i32)>;
}

struct OsrmStreamingProvider { /* ... */ }
struct EuclideanProvider { /* for testing */ }
```

### GPU side (WGSL)

Two new buffers in the megakernel:
- `@binding(17) sparse_neighbours: array<u32>`  — N×K, neighbour ids
- `@binding(18) sparse_distances: array<i32>`   — N×K, parallel distances

Change in `dist_at(a, b)`:
- If `coord_mode == 2`: linear scan a's K-list, fallback Euclidean
- (mode==0: matrix, mode==1: pure Euclidean, mode==2: sparse + fallback)

### Pipeline architecture

```
[OSRM] ───stream─→ [SparseCache builder] ───→ [GPU: upload coords + sparse]
                            │
                            └─ trigger LS start at warmup_threshold
                                            │
                                            ▼
                              [megakernel LS] ──→ Solution
```

## Tradeoffs / risks

1. **Cache miss rate in 2-opt**: unknown how often LS tries edges that are
   not K-neighbors. Likely <5% since granular eliminates distant candidates,
   but must be measured.
2. **OSRM throughput**: how fast can OSRM handle N×K table calls? Typically
   1000 points/sec per core → 50s for N=50K. Acceptable.
3. **Older nodes must update their K-list**: when a new point arrives
   that is closer than their K-th neighbor. Requires sorted insertion.
   Amortized O(N×K log K) total — manageable.
4. **Asymmetry**: OSRM distances are not symmetric (one-way streets).
   Sparse cache should store both directions separately — doubles memory but
   still 1000× less than dense.
